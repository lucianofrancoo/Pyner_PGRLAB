# PYNER MINER - BioProject Analyzer Architecture

## 1. Overview
The **BioProject Analyzer** is a specialized Phase 3 module in the PYNER MINER pipeline. It is designed to evaluate the **usability of raw experimental datasets** (BioProjects/SRA) rather than evaluating the relevance of published scientific papers.

This module actively decouples the analysis of BioProjects from the PubMed workflow, recognizing that a BioProject is a hierarchical container of raw data (Gigabytes of FASTQ files, distinct BioSamples, and multiple sequencing runs), not just a block of text.

## 2. Core Philosophy & Design Principles
*   **Dataset Usability over Paper Relevance:** The primary goal is to answer: *"Are the raw data files in this project technically and biologically useful for the user's query?"*
*   **Granularity at the Experiment/Run Level:** A single BioProject often contains multiple distinct experiments (e.g., wild-type vs. mutant, control vs. drought stress). The analyzer avoids treating the BioProject as a homogeneous unit. It flattens the hierarchy into individual rows for each SRA Experiment or Run whenever possible.
*   **Deterministic Metadata First, AI Second (Hybrid Scoring):**
    *   Technical metadata (Library Strategy, Platform, Instrument, Layout) and Biological metadata (Tissue, Condition, Genotype) are extracted deterministically directly from the BioSample and SRA metadata trees.
    *   The LLM (Ollama) is only used for high-level synthesis: judging overall experimental design, identifying major limitations, and providing a final usability classification.
*   **Organism and Domain Agnostic:** The system relies on generic metadata fields, ensuring it works seamlessly for plants, animals, microbes, or mixed systems.

## 3. Architecture Components

### A. The Orchestrator (`bioproject_analyzer.py`)
This script serves as the main entry point when PYNER MINER runs in `-d bioproject -m pro` mode.
1.  **Input:** Reads the structured JSON output generated by `boolean_fetcher_integrated.py` during Phase 2 (Cascade Search).
2.  **BioProjectNormalizer:** A purpose-built class that parses the complex, nested JSON (BioProject -> BioSample -> SRA Experiment -> SRA Run).
    *   *Attribute Flattening:* It actively scans BioSample attributes for keywords like 'tissue', 'stress', 'treatment', 'genotype' to extract precise biological context.
    *   *Volume Estimation:* Calculates the estimated data volume in GB based on SRA run sizes.
3.  **Heuristic Prefilter:** Evaluates the project deterministically (e.g., "Does it actually have SRA runs?", "Does the library strategy match the query intent?", "Does it have linked PubMed IDs?").
4.  **Output:** Generates a flat, robust TSV file (`bioproject_analysis_YYYYMMDD_HHMMSS.tsv`) containing 32 carefully selected columns, where each row represents a usable experimental unit (Run or Experiment).

### B. The AI Evaluator (`ollama_client.py` - BioProject Extension)
The existing Ollama client was extended with two new methods specifically for datasets: `analyze_bioproject` and `_build_bioproject_prompt`.
*   **The Prompt:** Acts as a "bioinformatics data curator." It receives the user's query, the project description, and a compacted preview of the SRA hierarchy (strategies, instruments).
*   **The Output Schema:** The LLM is restricted to returning a precise JSON containing:
    *   `relevance_score_0_10`: A 0-10 score of data usability.
    *   `experimental_design_summary`: A 2-3 sentence summary of the actual data gathered.
    *   `primary_use_case`: What the dataset is best suited for.
    *   `major_limitations`: Missing controls, poor metadata, etc.
    *   `recommended_for_query`: "Yes", "No", or "Tangential".
    *   `confidence_explanation`: Why this score was given.

### C. Pipeline Integration (`pyner_miner.sh`)
The Bash wrapper cleanly routes the workflow based on the database selected:
```bash
elif [ "$DATABASE" = "bioproject" ]; then
    # Phase 2: Fetch BioProject data (JSON generation)
    # ...
    if [ "$MODE" = "pro" ]; then
        # Phase 3: Route to the specialized BioProject Analyzer
        python3 bioproject_analyzer.py "$FETCH_JSON" "$ANALYSIS_TSV"
    fi
fi
```

## 4. Output TSV Structure (32 Columns)
The generated TSV provides a comprehensive, at-a-glance view of the experimental data landscape. Key column categories include:
*   **Identifiers:** `BioProject_ID`, `BioSample_ID`, `Experiment_ID`, `Run_ID`, `Linked_PMIDs`.
*   **Project Context:** `Title`, `Organism`, `Project_Type`.
*   **Technical Metrics:** `Library_Strategy`, `Library_Source`, `Platform`, `Instrument`, `Layout`, `Total_Samples`, `Estimated_Volume_GB`.
*   **Biological Context:** `Tissue`, `Condition`, `Treatment`, `Timepoint`, `Genotype`.
*   **AI Usability Assessment:** `Relevance_Score`, `Recommended_For_Query`, `Experimental_Design_Summary`, `Primary_Use_Case`, `Major_Limitations`, `Metadata_Completeness_Score`.
*   **Flags:** Internal markers (e.g., `project_only_no_sra`, `run_level_data`) describing the depth of available data.

## 5. Future Scalability
*   **Linking:** The `Linked_PMIDs` currently relies on explicit NCBI linkages. Future modules can implement text-mining on PMC full-texts to find "orphan" BioProjects mentioned in methodology sections (adding flags like `textual_accession_match`).
*   **Scoring:** The heuristic prefilter can be evolved into a machine learning classification model to decide whether to skip the LLM call entirely for irrelevant projects, saving compute time.
